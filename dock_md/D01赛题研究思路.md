# D01赛题研究思路

## 1. 目标与基本判断

D01赛题是一个“路径规划 + 生存博弈 + 收集收益最大化”的强化学习任务。智能体控制鲁班七号在 `128x128` 栅格地图中行动，需要在默认 `1000` 步内尽量存活，同时尽可能多收集宝箱和加速 buff。

最终评分为：

```text
总分 = 存活步数 * 1.5 + 宝箱数 * 100
```

因此研究目标不能只看训练 reward，而要直接围绕比赛 score 展开：

- 第一优先级：降低被怪物抓住的概率，提高稳定存活步数。
- 第二优先级：在安全边界内提高宝箱和 buff 收集效率。
- 第三优先级：提高 10 张开放地图上的泛化能力，避免只在少数地图高分。
- 第四优先级：利用闪现、路径评估和地图信息继续冲高上限。

当前代码中有两个智能体目录：

- `agent_ppo`：完整可运行 PPO 基线，但只使用 8 个移动动作，没有使用闪现，特征和奖励都比较简单。
- `agent_diy`：DIY 空模板，模型、算法、训练流程、样本处理、预测和保存加载接口基本都未实现。

因此后续研究不应直接在空的 `agent_diy` 上做复杂策略，而应先把 `agent_ppo` 作为可运行参照，逐步把 `agent_diy` 建成可训练、可评估、可迭代的自研方案。

推荐总路线：

```text
agent_ppo 作为基准
        ↓
agent_diy 复刻可运行 PPO 链路
        ↓
DIY 追平 8 动作基线
        ↓
升级 16 动作，接入闪现
        ↓
加入危险评估，提高存活
        ↓
加入宝箱 / buff 路径收益，提高得分
        ↓
多地图泛化训练和模型筛选
        ↓
高上限实验
```

## 2. 当前基线分析

### 2.1 agent_ppo 的优势

`agent_ppo` 已经具备完整训练链路：

- `agent.py`：实现了 `predict`、`exploit`、`learn`、`save_model`、`load_model`、`observation_process`、`action_process`。
- `algorithm/algorithm.py`：实现了 PPO clipped loss、value loss、entropy loss、masked softmax 和梯度更新。
- `feature/definition.py`：定义了 `ObsData`、`ActData`、`SampleData`，并实现 GAE。
- `feature/preprocessor.py`：实现 40 维基础特征和简单生存奖励。
- `workflow/train_workflow.py`：实现环境交互、样本收集、样本发送、模型保存和监控上报。

它适合作为参照组，用来判断自研方案是否真的提分。

### 2.2 agent_ppo 的短板

`agent_ppo` 当前短板明显：

- 动作空间只有 `8` 维，只使用普通移动动作 `0-7`，没有使用闪现动作 `8-15`。
- 特征只有 `40` 维，只截取了 `4x4` 局部地图，没有充分利用环境提供的 `21x21` 视野。
- 奖励主要是每步生存奖励和远离怪物奖励，缺少宝箱、buff、闪现、路径质量、探索等目标。
- 模型是简单 MLP，无法直接处理完整局部地图结构。
- 缺少逐地图评估、失败率、宝箱数、buff 数、闪现效果等研究指标。

这些短板也是后续提分的主要方向。

### 2.3 agent_diy 的现状

`agent_diy` 目前是空模板：

- `agent.py` 中核心接口基本为 `pass`。
- `algorithm.py` 没有训练逻辑。
- `model.py` 没有网络结构。
- `definition.py` 中 `reward_shaping` 和 `sample_process` 未实现。
- `train_workflow.py` 只有占位代码。
- 当前配置为 `FEATURE_VECTOR_SHAPE = (153,)`、`ACTION_SHAPE = (8,)`，与后续 16 动作目标不一致。

因此 `agent_diy` 的第一阶段目标不是提分，而是先跑通。

## 3. 阶段化研究路线

- 确认 `train_test.py`、模型保存、模型加载、监控上报都可正常使用。
- 建立实验记录表，所有后续改动都要与该基线比较。

- 已有一份可复现实验记录。

### 阶段 1：让 agent_diy 跑通

目标：把 `agent_diy` 从空模板变成完整可训练智能体。

工作内容：

- 参考 `agent_ppo` 实现 `agent_diy` 的完整 PPO 训练链路。
- 先保持简单 MLP 和基础特征，不急着引入复杂策略。
- 实现 `predict`、`exploit`、`learn`、`save_model`、`load_model`。
- 实现 `observation_process`、`action_process`。
- 实现 `SampleData`、GAE、PPO loss 和 masked softmax。
- 实现训练 workflow：reset、predict、step、reward、sample、send_sample_data、save_model。
- 修改 `train_test.py` 中 `algorithm_name = "diy"`，验证 DIY 可训练。

本阶段建议仍使用 8 动作，目的是降低变量数量，先排除训练链路问题。

进入下一阶段条件：

- `agent_diy` 可以完成短训练。
- loss、reward、episode 指标正常上报。
- 模型可以保存和加载。
- 不出现样本维度、动作维度、模型参数加载错误。

### 阶段 2：agent_diy 追平 agent_ppo

目标：证明 DIY 链路正确，不低于原始 PPO 基线。

工作内容：

- 在 `agent_diy` 中复现 `agent_ppo` 的基础 40 维特征。
- 使用相近 PPO 超参数、奖励设计和模型结构。
- 保持 8 动作，不引入闪现。
- 对比相同训练时间下的 score、steps、fail_rate。
- 优先使用同一套日志解析脚本重新统计 `agent_ppo` 和 `agent_diy`，不要用阶段 0 手工监控近似值直接做严格比较。

本阶段不追求创新，只追求可控复现。

进入下一阶段条件：

- DIY 平均分接近或超过 `agent_ppo`。
- DIY 平均存活步数不明显低于 `agent_ppo`。
- 训练曲线没有明显异常震荡。

如果 DIY 明显低于 PPO，应先排查：

- `legal_action` mask 是否和 logits 对齐。
- `prob` 是否记录旧策略概率。
- GAE 中 `next_value` 是否正确。
- `done` 和终局奖励是否正确。
- `exploit` 是否使用确定性动作。

### 阶段 3：升级到 16 动作，接入闪现

目标：让智能体具备使用闪现的能力。

工作内容：

- 将动作空间从 8 维改为 16 维。
- `Config.ACTION_SHAPE` 或等价配置改为 `(16,)`。
- `SampleData` 中动作概率和合法动作维度改为 16。
- 模型 actor 输出改为 16 维 logits。
- `legal_action` 使用环境返回的完整 16 维 mask。
- `predict` 支持采样 `0-15`。
- `exploit` 支持选择 `0-15` 中合法动作。
- `action_process` 直接返回环境动作编号。

初期不要让闪现奖励太复杂，只需要避免明显错误：

- 原地闪现或无意义闪现给轻微负奖。
- 危险时闪现后远离怪物给正奖。
- 闪现路径收集宝箱或 buff 给正奖。

进入下一阶段条件：

- 训练过程不因 16 动作维度报错。
- 评估中能输出合法闪现动作。
- 闪现使用次数不为 0，也不出现开局连续乱闪现。
- 存活步数不低于 8 动作版本太多。

### 阶段 4：加入危险评估，优先提高存活

目标：降低失败率，提高平均存活步数和最低地图表现。

工作内容：

- 加入怪物相关特征：
  - 怪物是否可见。
  - 怪物相对方向。
  - 怪物距离桶。
  - 怪物速度。
  - 第二只怪物是否已出现。
  - 怪物是否已加速。
- 加入候选动作安全评估：
  - 执行动作后的英雄位置估计。
  - 到最近怪物的距离变化。
  - 是否进入怪物 `3x3` 危险区。
  - 是否靠近死路或障碍密集区。
  - 闪现落点是否安全。
- 将 16 个动作的安全分作为模型输入，而不是直接替模型做全部决策。

奖励设计重点：

| 奖励项 | 作用 |
| :--- | :--- |
| 每步生存小正奖 | 鼓励持续存活 |
| 被抓强负奖 | 明确终局失败代价 |
| 进入危险区负奖 | 提前避免死亡 |
| 远离最近怪物正奖 | 引导逃生方向 |
| 撞墙 / 原地无效移动负奖 | 减少无效动作 |
| 危险中有效闪现正奖 | 学会用闪现逃命 |

进入下一阶段条件：

- fail_rate 明显下降。
- avg_steps 提升。
- map_min_score 提升。
- 没有明显“只逃跑不拿宝箱”的极端行为，或者该问题被记录到下一阶段处理。

### 阶段 5：加入宝箱和 buff 收益

目标：在保持生存的基础上提高收集收益。

工作内容：

- 加入宝箱特征：
  - 可见宝箱数量。
  - 最近宝箱距离、方向、可达性。
  - 多个宝箱的收益排序。
  - 宝箱路径上的危险程度。
- 加入 buff 特征：
  - 可见 buff 数量。
  - 最近 buff 距离、方向、可达性。
  - 当前是否处于加速状态。
  - buff 剩余时间。
- 加入局部路径规划：
  - 在 `21x21` 视野内对宝箱、buff、安全区做 BFS 或 A*。
  - 计算候选动作到目标的路径进展。
  - 计算收益 / 风险比。

奖励设计重点：

| 奖励项 | 作用 |
| :--- | :--- |
| 拾取宝箱强正奖 | 对齐最终 score |
| 拾取 buff 中等奖 | 鼓励获取机动性 |
| 靠近高价值目标正奖 | 提高收集效率 |
| 远离目标负奖 | 减少无意义逃跑 |
| 重复绕圈负奖 | 减少局部震荡 |
| 为宝箱冒死负奖 | 防止过度贪心 |

进入下一阶段条件：

- treasure_avg 提升。
- total_score 提升。
- avg_steps 和 fail_rate 不明显恶化。
- 出现“安全时主动找宝箱，危险时优先逃生”的行为。

### 阶段 6：多地图泛化

目标：提高隐藏地图上的泛化可能性。

工作内容：

- 训练时使用全部 10 张开放地图。
- 建议开启或模拟 `map_random = true`。
- 评估必须按地图分别统计，而不是只看总平均。
- 对低分地图做单独行为分析。
- 尝试随机化环境参数：
  - `monster_interval = -1`
  - `monster_speedup = -1`
  - 不同 `treasure_count`
  - 不同 `buff_count`
- 做课程训练：
  - 先低难度学存活。
  - 再默认配置学收集。
  - 最后随机参数增强鲁棒性。

模型选择标准：

```text
优先选择：平均分高、最低地图分不低、失败率低、地图间方差小的模型。
不选择：只在少数地图高分、但最低地图很差的模型。
```

进入下一阶段条件：

- 10 张开放地图表现更均衡。
- map_min_score 提升。
- map_score_std 下降。
- 默认配置和随机配置下都能稳定运行。

### 阶段 7：高上限实验

目标：在稳定策略基础上进一步冲分。

候选方向：

- 使用完整 `21x21` 地图 CNN，而不是只用手工向量。
- 加入短历史轨迹特征，例如最近 5 步位置、动作、怪物方向。
- 加入地图记忆，记录已探索区域、最近访问位置、疑似安全区。
- 多头价值函数，将生存价值和收集价值分开估计。
- 引入动作先验，将规则安全分作为 logits bias。
- 分阶段策略：前期主动收集，中期平衡，后期保命。
- 对闪现做专项策略头，区分移动决策和技能决策。

实验原则：

- 每次只改一个关键因素。
- 每个实验保留配置、训练时间、模型编号、评估结果。
- 不以单次最高分判断好坏，至少看两轮评估。
- 如果平均分提高但失败率明显升高，要谨慎保留。

## 4. 推荐指标体系

训练中至少记录以下指标：

| 指标 | 用途 |
| :--- | :--- |
| reward | 观察训练奖励趋势 |
| total_loss | 观察训练是否稳定 |
| policy_loss | 判断策略更新幅度 |
| value_loss | 判断价值函数是否学习 |
| entropy_loss | 判断探索是否过早塌缩 |
| episode_steps | 观察存活能力 |
| episode_score | 直接观察比赛分数 |
| treasures_collected | 观察收集能力 |
| collected_buff | 观察 buff 利用 |
| flash_count | 观察闪现使用频率 |
| fail_rate | 观察死亡风险 |

评估时建议按如下格式记录：

| 实验编号 | 模型 | 地图 | 平均分 | 平均步数 | 失败率 | 宝箱 | buff | 闪现 | 备注 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| exp001 | diy_8act_base | map_1 | | | | | | | |
| exp001 | diy_8act_base | map_2 | | | | | | | |

实验记录要优先关注：

- 哪些地图经常失败。
- 失败通常发生在第几步。
- 失败前是否有闪现可用。
- 是否因为贪宝箱死亡。
- 是否出现反复撞墙、绕圈、远离宝箱等行为。

## 5. 关键实现决策

### 5.1 是否继续使用 PPO

建议继续使用 PPO 作为第一主线。

理由：

- `agent_ppo` 已经有完整可参考实现。
- PPO 对 on-policy 策略较稳定。
- 当前任务动作空间小，PPO 足够作为强基线。
- 单卡资源下，优先改特征和奖励比更换算法更划算。

不建议一开始切换到复杂 off-policy 算法。等 PPO 强基线稳定后，可以再尝试 DQN、SAC 离散版或其他方法。

### 5.2 是否一开始就使用 CNN

不建议在 `agent_diy` 刚跑通时立即使用 CNN。

建议顺序：

```text
MLP + 手工特征
        ↓
MLP + 路径/危险规则特征
        ↓
CNN 处理 21x21 map_info + MLP 处理向量特征
```

这样更容易定位收益来源。

### 5.3 是否一开始就使用 16 动作

建议在 DIY 追平 PPO 后再升级 16 动作。

原因：

- `agent_diy` 目前空模板，先做 16 动作会增加排错难度。
- `agent_ppo` 是 8 动作，可作为追平目标。
- 追平后升级 16 动作，可以单独衡量闪现带来的收益。

但一旦进入 16 动作阶段，就不要再频繁切回 8 动作，否则模型、样本和评估结果不可比。

## 6. 风险与排查清单

### 6.1 训练链路风险

常见问题：

- `SampleData` 维度和模型输入输出不一致。
- `legal_action` 维度与 logits 不一致。
- `prob` 记录的不是旧策略概率，导致 PPO ratio 错误。
- `next_value` 未正确填充，GAE 失真。
- `done` 处理错误，终局奖励错位。
- `exploit` 使用随机采样，导致评估不稳定。

排查顺序：

```text
先跑 train_test 短流程
再检查样本维度
再检查 action/prob/mask
再检查 reward 和 done
再看 loss 和评估行为
```

### 6.2 奖励设计风险

常见问题：

- 奖励过大导致 value loss 爆炸。
- 生存奖励太强，智能体只逃跑不拿宝箱。
- 宝箱奖励太强，智能体冒死贪宝箱。
- 距离塑形太强，智能体在怪物边缘来回震荡。
- 闪现奖励不当，导致开局乱闪现。

处理原则：

- 每次只加一个奖励项。
- 奖励值先小后大。
- 每次改奖励都观察行为，不只看曲线。
- 最终以 `total_score` 和 `fail_rate` 判断。

### 6.3 泛化风险

常见问题：

- 单地图训练高分，多地图掉分。
- 只记住固定宝箱路线。
- 隐藏地图障碍结构不同导致失效。
- 某些地图最低分极差，但平均分被高分地图掩盖。

处理原则：

- 每轮评估都按地图拆分。
- 保留最低地图分和分数标准差。
- 尽早使用多地图训练。
- 后期加入环境参数随机化。

## 7. 推荐执行顺序

短期执行：

1. 跑 `agent_ppo`，得到基础分数和日志。
2. 在 `agent_diy` 中复刻 PPO 训练链路。
3. 用 8 动作 DIY 追平 `agent_ppo`。
4. 建立统一评估脚本和实验记录表。

中期执行：

1. 将 DIY 升级为 16 动作。
2. 加入闪现基础策略。
3. 加入怪物危险评估。
4. 优化生存奖励。
5. 对比 8 动作和 16 动作收益。

后期执行：

1. 加入宝箱和 buff 路径收益。
2. 引入局部 BFS/A* 特征。
3. 做多地图泛化训练。
4. 做模型筛选和消融实验。
5. 尝试 CNN、历史轨迹、多头价值等高上限方向。

## 8. 最终研究原则

- 先让系统可运行，再谈提分。
- 先追平基线，再做创新。
- 先提高存活，再提高收益。
- 先做强特征和奖励，再堆大模型。
- 每次只改一个关键因素。
- 所有结论必须通过多地图评估验证。
- 不以训练 reward 作为唯一指标，最终以比赛 score、失败率和泛化表现为准。
