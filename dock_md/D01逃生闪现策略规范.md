# D01逃生闪现策略规范

## 0. 文档定位

本文档用于定义一套**独立于具体代码框架**的“逃生闪现策略规范”，供后续接入任意版本的智能体策略。

当前阶段只研究**逃生类闪现**，不研究以下内容：

- 安全区赶路闪现
- 宝箱 / buff 收益型闪现
- 纯奖励塑形驱动的闪现学习
- 与具体 PPO / rule / hybrid 代码结构强绑定的实现细节

本文档后续所有更新、实验记录和版本结论，统一追加在本文件中维护。

## 2. 规则基础

本策略基于开发指南中的环境规则成立：

- 闪现方向固定为 8 个方向，对应动作 `8-15`
- 直线闪现距离为 `10` 格，斜向闪现距离为 `8` 格
- 闪现落点不是任意指定点，而是该方向范围内**最远的可通行格**
- 若范围内无可通行格子，则原地闪现且消耗冷却
- 闪现可以跨越障碍物，但最终只能落在可通行位置
- 任务失败条件为：英雄进入怪物中心 `3x3` 范围，即英雄与怪物距离小于等于 `1` 格

因此，逃生闪现的核心不是“是否使用了闪现”，而是：

1. 闪后是否离怪物更远
2. 闪后是否脱离当前 danger / near-threat 状态
3. 若闪前已进入怪物 `3x3` 威胁区，闪后是否离开该威胁区
4. 闪后是否来到更开阔、更安全的位置

## 3. 策略目标

### 3.1 主目标

仅在危险场景下，使用闪现完成以下逃生动作：

- **贴身脱怪**：怪物已逼近，普通移动无法立即明显脱险
- **穿墙脱怪**：怪物与英雄同侧逼近，闪现跨障碍换边
- **过窄口 / 过拐角脱怪**：从受限空间快速切换到更开阔区域

当前实验已经验证：  
V1 的主能力不是“进入 true threat 后再闪出”，而是更偏向于：

```text
在 near-threat / danger 阶段提前闪现
→ 拉开怪物距离
→ 离开 danger
→ 进入更开阔、更安全的位置
```

因此，当前版本应优先被理解为：

**预防式逃生闪现策略**

### 3.2 次目标

- 不为了提高闪现次数而使用闪现
- 不在安全区无收益乱闪
- 不让闪现把英雄送进死角、障碍包夹区、窄口深处
- 不为了强行触发 `leave_threat` 而推迟闪现时机

## 4. 输入定义

本策略规范只定义所需信息的语义，不绑定代码字段名。

### 4.2 推荐输入

| 输入项 | 含义 |
| :--- | :--- |
| monster_pos | 怪物精确位置 |
| monster_speed | 怪物速度 |
| second_monster_pos | 第二怪物位置 |

| current_move_best_score | 当前最佳普通移动的安全性评分 |

### 4.3 派生输入

策略需要从原始输入中派生出以下量：

- 最近怪物距离
- 两只怪物中的最小安全裕量
- 当前是否处于怪物 `3x3` 威胁区
- 当前是否处于“近危险区”
- 每个闪现方向的实际落点
- 每个落点周围的可通行面积 / 逃生自由度
- 每个候选落点是否跨障碍换边
- 每个候选落点是否进入死角 / 窄口深处

## 5. 关键状态定义

### 5.1 威胁区

**威胁区**定义为英雄落在某个怪物中心 `3x3` 范围内。  
等价地，在栅格语义上可按如下条件判定：

```text
max(abs(hero_x - monster_x), abs(hero_z - monster_z)) <= 1
```

满足上式时，视为已进入怪物直接捕获危险区。

### 5.2 近危险区与危险态

V1.2 推荐将与 flash 触发相关的状态分成三层：

#### A. true threat

```text
max(abs(hero_x - monster_x), abs(hero_z - monster_z)) <= 1
```

表示英雄已经进入怪物 `3x3` 直接威胁区。

#### B. near-threat

推荐按如下语义实现：

```text
当前尚未进入 true threat，
但下一步极易进入 threat，
或普通移动的安全裕量已经明显不足
```

若采用简化栅格判定，可先从以下口径起步：

```text
max(abs(hero_x - monster_x), abs(hero_z - monster_z)) == 2
```

#### C. danger

danger 是 flash planner 的**触发态**，可以宽于 near-threat。  
在当前验证中，flash 主要发生在 near-threat / danger 阶段，而不是 true threat 阶段。

因此：

- `true threat` 用于定义**最严格的危险区**
- `near-threat` 用于定义**即将进入 threat 的高压区**
- `danger` 用于定义**当前 flash planner 的主触发态**

V1.2 的实验结论表明：

```text
当前有效 flash 几乎都发生在 near-threat / danger 阶段，
并在该阶段完成 leave_danger，
而不是在 true threat 中再 leave_threat。
```

### 5.3 实际闪现落点

对任一闪现方向，按环境规则生成实际落点：

1. 先确定该方向的理论最远格
2. 从最远格向近处回退
3. 找到该方向范围内最远的可通行格
4. 若整个范围均不可通行，则该方向视为原地闪现

策略决策必须使用**实际落点**评分，不允许只用理论终点评分。

### 5.4 开阔度

V1 中“开阔度”可用以下任一近似实现：

- 落点周围 `3x3` 或 `5x5` 范围内可通行格数量
- 从落点出发，`2-3` 步内可到达的可通行格数量
- 落点处 8 邻域中可继续移动的方向数

推荐优先使用：

```text
2-3 步内可到达的可通行格数量
```

因为它更接近“闪后还能不能继续逃”。

### 5.5 穿墙成功

满足以下条件时，记为**穿墙成功**：

1. 闪现路径上存在障碍物
2. 实际落点与起点分处障碍的不同侧
3. 落点仍为可通行位置

### 5.6 窄口 / 拐角脱离成功

满足以下条件之一时，可记为**窄口 / 拐角脱离成功**：

- 落点周围的开阔度明显高于起点
- 落点可继续移动方向数明显增加
- 起点附近属于窄通道 / 障碍夹角，而落点不再属于该受限区域

### 5.7 状态转移指标定义

为避免把不同层次的“有效”混为一谈，V1.2 明确区分以下量：

#### `flash_pre_in_threat`

flash 发生前，英雄已经处于 true threat。

#### `flash_pre_in_near_threat`

flash 发生前，英雄不在 true threat，但处于 near-threat。

#### `flash_leave_danger`

flash 前属于 danger / near-threat，flash 后已不再属于该 danger 状态。

#### `flash_leave_threat`

flash 前已进入 true threat，flash 后离开 true threat。

#### `escape_effective_flash`

当前版本定义为更宽泛的“逃生有效”：

- 闪前处于 danger / near-threat
- 闪后安全性显著改善
- 通常伴随距离增益、最小安全裕量增益或开阔度增益

当前实验结论：

```text
escape_effective ≠ leave_threat
当前版本大多数 escape_effective 来自 near-threat 阶段的 leave_danger
```

## 6. 触发条件

V1 规定：**默认不评估闪现**，只有进入危险决策状态时才开启闪现评分。

### 6.1 强制评估闪现

满足以下任一条件时，必须评估闪现：

1. 当前已进入怪物 `3x3` 威胁区
2. 当前处于 near-threat，且最佳普通移动无法明显脱离 danger
3. 两只怪物夹击导致普通移动后最小安全裕量仍很低

### 6.2 建议评估闪现

满足以下条件时，建议评估闪现：

1. 最近怪物很近，且普通移动距离增益很小
2. 某个闪现方向可明显实现穿墙换边
3. 某个闪现方向可从窄口 / 拐角切换到更开阔区域
4. 当前虽未进 true threat，但已明显处于 near-threat / danger 状态

### 6.3 不评估闪现

满足以下任一条件时，直接不进入闪现评估：

1. 闪现冷却中
2. 当前不危险，且最佳普通移动已经具备明确安全解
3. 本帧没有任何可接受的实际闪现落点

## 7. 一票否决条件

以下情况任一成立，则该闪现方向直接判为不可选：

1. **原地闪现**：实际落点与当前点相同
2. **短闪无收益**：位移很短，且没有离开 danger、没有明显增大怪物距离
3. **仍在威胁区 / 危险区**：闪后仍处于 true threat 或 danger，且距离改善不明显
4. **向怪物靠近**：闪后与最近怪物的距离不增反降
5. **落入死角**：落点周围可通行空间明显不足
6. **进入障碍包夹区**：后续可逃方向显著少于当前点
7. **把自己送向第二怪物**：对最近怪物变远，但对第二怪物明显变近，且整体最小安全裕量更差
8. **安全区无必要闪现**：当前不危险，且本次闪现没有明确逃生收益

## 8. 候选动作生成

### 8.1 闪现候选

生成 8 个闪现候选方向：

- `8` 右
- `9` 右上
- `10` 上
- `11` 左上
- `12` 左
- `13` 左下
- `14` 下
- `15` 右下

对每个方向必须计算：

1. 理论终点
2. 实际落点
3. 位移长度
4. 是否穿墙
5. 落点与两只怪物的距离变化
6. 落点处开阔度
7. 是否落入死角 / 窄口深处

### 8.2 普通移动候选

同时生成 8 个普通移动候选，用于和闪现候选做比较。

注意：  
V1 不是“闪现内部选最优方向”就直接执行，而是要比较：

```text
最佳闪现  vs  最佳普通移动
```

## 9. 方向评分函数

V1 评分函数只服务于**逃生规划**，不加入宝箱 / buff 收益。

### 9.1 必算评分项

| 评分项 | 含义 | 方向 |
| :--- | :--- | :--- |
| distance_gain_score | 闪后与最近怪物距离是否明显增加 | 越大越好 |
| leave_danger_score | 是否脱离当前 danger / near-threat | 脱离为强正项 |
| leave_threat_score | 若闪前在 true threat，是否脱离怪物 `3x3` 威胁区 | 仅在 threat 中生效 |
| min_margin_score | 对两只怪物中的最小安全裕量是否提升 | 越大越好 |
| openness_score | 落点周围是否更开阔 | 越大越好 |
| wall_cross_score | 是否成功穿墙换边 | 成功为正项 |
| choke_escape_score | 是否成功离开窄口 / 拐角受限区域 | 成功为正项 |
| dead_end_penalty | 落点是否更接近死角 | 越大越差 |
| invalid_flash_penalty | 原地闪、短闪、坏落点闪 | 越大越差 |

### 9.2 V1 推荐优先级

优先级从高到低：

1. **是否脱离当前 danger / near-threat**
2. **与最近怪物距离是否显著增加**
3. **落点是否更安全、更开阔**
4. **是否完成穿墙换边**
5. **是否完成窄口 / 拐角脱离**
6. **若闪前已在 true threat，是否完成 leave_threat**

资源收益不进入 V1 评分。

### 9.3 V1 推荐总分形式

可采用如下抽象形式：

```text
flash_score =
  a * leave_danger_score
+ b * distance_gain_score
+ c * min_margin_score
+ d * openness_score
+ e * wall_cross_score
+ f * choke_escape_score
+ g * leave_threat_score
- h * dead_end_penalty
- i * invalid_flash_penalty
```

V1 推荐相对权重关系：

```text
a >= b >= c >= d
e 与 f 为中等奖励
g 仅在 true threat 中提供额外加分
h 与 i 为强惩罚
```

即：

- “脱离 danger / near-threat”必须是最高优先级
- “坏落点 / 原地闪 / 送死闪”必须强惩罚
- “穿墙成功”只有在安全性成立时才算加分，不能单独保送

### 9.4 V1 默认启发式权重

若需要一版可直接落地的初始权重，可从以下关系起步：

```text
leave_danger_score      = +5 / -4
distance_gain_score     = 按距离增量线性计分，裁剪到 [-4, +4]
min_margin_score        = 裁剪到 [-3, +3]
openness_score          = 裁剪到 [-3, +3]
wall_cross_score        = +2
choke_escape_score      = +2
leave_threat_score      = +2（仅 true threat 内生效）
dead_end_penalty        = +3
invalid_flash_penalty   = +6
```

说明：

- 这是 V1 的默认启发式，不是最终固定参数
- 后续实验只允许小步调整，不应同时大改全部权重

## 10. 普通移动与闪现的比较原则

V1 不追求“尽量多闪”，而追求“闪现只在明显优于普通移动时接管”。

### 10.1 必须满足的比较原则

只有当以下条件成立时，闪现才允许胜出：

1. 最佳闪现方向没有触发一票否决
2. 最佳闪现方向的安全性明显优于最佳普通移动
3. 或者最佳普通移动无法脱离威胁区，而最佳闪现可以

### 10.2 推荐决策规则

```text
若最佳普通移动已经能脱离 danger，且安全裕量充足，则不闪现
若最佳普通移动不能脱离 danger，而最佳闪现可以，则优先闪现
若二者都能脱离 danger，则选择更高安全裕量、更高开阔度者
若最佳闪现仅略优于最佳普通移动，不应为了提高闪现利用率而强行闪现
```

### 10.3 V1 推荐阈值

若需要设定最小覆盖阈值，推荐：

```text
best_flash_score >= best_move_score + delta
```

其中 `delta` 建议从较小正数开始，例如：

```text
delta = 2
```

若处于**已进入威胁区**的紧急状态，可放宽 `delta`，优先能脱险即可。
若当前主要处于 near-threat 而未进入 true threat，则优先比较 `leave_danger` 和安全裕量提升。

## 11. 三类逃生闪现的专用判定

### 11.1 贴身脱怪

适用条件：

- 最近怪物已进入 danger / near-threat
- 普通移动无法立即明显拉开距离

当前验证表明：  
贴身脱怪的主实现形态不是“进入 true threat 后再闪”，而是：

```text
在 near-threat 阶段提前闪现
→ 拉开距离
→ leave_danger
```

重点评分：

- `leave_danger_score`
- `distance_gain_score`
- `min_margin_score`
- `openness_score`

### 11.2 穿墙脱怪

适用条件：

- 怪物与英雄当前在障碍同侧
- 某个闪现方向可跨障碍换边

重点评分：

- `wall_cross_score`
- `distance_gain_score`
- `leave_danger_score`
- `openness_score`

注意：

- 穿墙本身不等于安全
- 若墙后是死角或狭窄袋状区域，该方向仍应被否决或重罚

### 11.3 过窄口 / 过拐角

适用条件：

- 当前点位于窄通道、拐角、障碍边缘附近
- 某个闪现方向能切换到更大可通行区域

重点评分：

- `choke_escape_score`
- `openness_score`
- `min_margin_score`

当前实验中，该分支已经出现但强度弱于穿墙脱怪，应视为**正在形成中的辅助能力**。

## 14. 当前结论

V1 的核心立场如下：

1. 闪现不是常规移动替代，而是危险时的应急重定位动作
2. 当前已验证的主行为是：**在 near-threat / danger 阶段提前闪现，完成 leave_danger**
3. 当前尚未验证“在 true threat 中再稳定 leave_threat”是主能力
4. 穿墙脱怪已较成熟，过窄口 / 过拐角已开始出现，但仍弱于穿墙主分支
5. 若普通移动已能安全脱险，则不应强行使用闪现
6. 安全区无明确逃生收益的闪现，在 V1.2 中默认强抑制
7. `leave_threat = 0` 在当前主版本中不应直接视为 bug，需结合 `pre_in_threat / pre_in_near_threat / leave_danger` 一起解释

## 15. 推荐接入方式

V1.2 推荐作为**独立闪现决策层**接入强主策略，而不是整套主策略替换。

### 15.1 推荐架构

```text
强移动主策略
    ↓
危险 / near-threat 判断
    ↓
flash planner（是否闪、往哪闪）
    ↓
若最佳 flash 明显优于最佳 move，则覆盖为 flash
否则继续使用主策略 move
```

### 15.2 推荐理由

1. 当前已验证的能力主要集中在 flash 机制本身，而不是整套主策略上限
2. 强主策略通常已经具备较好的普通移动逃生能力
3. 当前 flash planner 的主要增益来自：
   - near-threat 阶段预防式脱险
   - 穿墙换边
   - 开阔度与安全裕量提升
4. 以模块方式接入，更容易保留主策略原有优势，同时补足闪现能力

### 15.3 当前不建议的接入方式

- 不建议用当前 flash 训练版本整体替换队友最强模型
- 不建议把当前 flash reward / checkpoint / 全量行为直接迁移过去
- 不建议把 `leave_threat` 作为当前主版本的唯一成功标准
## 16. 后续版本计划

V2 可考虑加入：

- 与普通移动共享的一致安全评分接口
- 双怪夹击下的更细安全裕量定义
- 资源路径收益作为次级加分项
- 闪现后 `3-5` 步短期可持续生存评分
- 更精细的窄口 / 拐角 / 障碍分区识别
- 若后续确有需求，再单开 “true threat 内 hard leave_threat” 分支实验

当前不建议在 V1 阶段加入：

- 宝箱 / buff 收益型闪现
- 大量奖励塑形参数联动
- 与具体训练框架强耦合的实现约束
