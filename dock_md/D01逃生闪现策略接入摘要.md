# D01逃生闪现策略接入摘要

## 0. 文档用途

本文档是 [D01逃生闪现策略规范.md](E:/Agent/codes/code/dock_md/D01逃生闪现策略规范.md) 的精简接入版。

用途只有一个：

**提供给负责接入队友最好版本的实现线程，快速理解当前已验证的闪现能力、能力边界和推荐接法。**

本文档不展开完整研究过程，不承载监控细节推导，也不替代主规范文档。

---

## 1. 一句话结论

当前已验证成功的，不是“进入怪物 `3x3` threat 后再硬闪出去”的闪现策略，而是：

**在 `near-threat / danger` 阶段提前触发的预防式脱险闪现策略。**

它的主要能力是：

- 怪物逼近时主动闪现拉开距离
- 大量利用穿墙换边脱险
- 多数情况下把状态从 `danger / near-threat` 带到更安全的位置

当前最准确的定位是：

**独立的预防式逃生闪现模块**

而不是：

**完整主策略替代品**

---

## 2. 当前已经验证成立的能力

### 2.1 已验证成立

1. **danger 主导触发**
   - 当前 flash 主要发生在 `danger / near-threat` 场景
   - 不再是安全区乱闪

2. **预防式脱险成立**
   - 主要模式是：在进入 `true threat` 前提前闪现
   - 从 `danger / near-threat` 状态转移到更安全状态

3. **穿墙脱怪已较成熟**
   - 当前最稳定、最强的 flash 能力就是 `wall_cross escape`

4. **闪后整体更远、更安全、更开阔**
   - 距离增益、最小安全裕量增益、开阔度增益都已经验证为正

5. **planner 主导已经基本成立**
   - 大多数实际 flash 已经可以由 planner 解释

### 2.2 初步成立但还不强

1. **过窄口 / 过拐角**
   - 已出现
   - 但强度和稳定性还弱于穿墙脱怪

### 2.3 当前未验证成立

1. **true threat 内的 leave_threat**
   - 目前实验显示：
     - `flash_pre_in_threat_count_sum = 0`
     - `flash_leave_threat_count_sum = 0`
   - 说明当前策略几乎不会等进入怪物 `3x3` threat 后再闪
   - 因此当前不能宣称“已验证 true threat 内硬脱险”

---

## 3. 对接入线程最重要的能力边界

### 3.1 应如何理解当前 flash 模块

当前模块应该被理解为：

```text
当怪物已经逼近、普通移动安全裕量明显不足时，
优先在 near-threat / danger 阶段用 flash 提前脱险，
并优先利用穿墙换边和开阔度提升来提高生存率。
```

### 3.2 不应如何理解当前 flash 模块

不要把当前模块理解成：

```text
已经进入怪物 3x3 threat 后，也能稳定把英雄闪出去的最终版硬脱险模块
```

当前数据不支持这个说法。

---

## 4. 推荐接入方式

### 4.1 总原则

**只接 flash 决策层，不替换队友的主移动策略。**

推荐架构：

```text
队友最好版本主策略
    负责普通移动 / 主体行为

+ 独立 flash planner
    只在 danger / near-threat 时介入
    决定：
    - 是否评估 flash
    - 8 个方向里哪个 flash 最优
    - flash 是否优于当前最佳普通移动
```

### 4.2 为什么这样接

原因很明确：

1. 当前实验真正验证成功的是 **flash 机制**
   - 不是整套主策略上限

2. 队友最好版本的主移动能力已经更强
   - 当前模块最有价值的是给它补一个“危险逼近时的预防式闪现”

3. 这样接入风险最小
   - 不会破坏队友主策略
   - 也便于做有无 flash planner 的 A/B 对比

### 4.3 不推荐的接法

不建议：

1. 直接拿当前训练出的整套策略替换队友最好版本
2. 直接整体并入当前 reward / checkpoint / 全量行为
3. 把当前版本包装成“已经验证 true threat leave_threat”的模块

## 5. 接入时真正需要实现的主干

实现线程真正需要抓住的只有下面这条主链：

```text
danger / near-threat 触发
-> 生成 8 个 flash 候选
-> 计算每个候选的实际落点
-> 按逃生评分函数打分
-> 与最佳普通移动比较
-> 只有当最佳 flash 明显更优时才执行 flash
```

### 5.1 触发条件

flash planner 只应在以下场景介入：

1. 当前处于 `danger / near-threat`
2. 普通移动已经不能提供足够安全裕量
3. 某些 flash 候选能明显实现：
   - 拉开怪物距离
   - 穿墙换边
   - 切到更开阔区域

在安全区默认不介入。

### 5.2 候选动作

只需要评估：

- 8 个普通移动方向
- 8 个 flash 方向（动作 `8-15`）

必须使用：

**实际闪现落点**

不能只按理论终点打分。

### 5.3 当前最重要的评分项

按重要性排序，当前实现应优先保留：

1. `leave_danger`
2. `distance_gain`
3. `min_margin_gain`
4. `openness_gain`
5. `wall_cross`
6. `choke_escape`

其中：

- `leave_danger` 是当前主目标
- `wall_cross` 是当前最成熟的地形能力
- `choke_escape` 可以保留，但不要把它当主收益来源

### 5.4 当前不应强依赖的评分项

当前不应把以下项作为主目标：

1. `leave_threat`
   - 当前主策略几乎不在 true threat 内触发 flash

2. 宝箱 / buff 收益
   - 当前模块不是资源 flash 模块

3. 全局赶路收益
   - 当前模块不是路径压缩模块

---

## 6. 当前实现要优先实现的三类 flash

### 6.1 贴身脱怪

当前更准确的实现目标不是：

“已经进 3x3 后再闪出去”

而是：

“怪物已逼近，普通移动安全裕量不足时，在 near-threat 阶段提前闪开”

### 6.2 穿墙脱怪

这是当前最值得保留的能力。

如果候选 flash 能：

- 跨过障碍
- 换到障碍另一侧
- 同时提高安全裕量或开阔度

则应高优先级考虑。

### 6.3 过窄口 / 过拐角

这类能力可以保留，但不要重压。

当前更合理的态度是：

- 保留该分支
- 给适度加分
- 不要为了追它而破坏穿墙脱怪和 leave_danger 主线

---

## 7. 当前接入后的验收重点

接入到队友最好版本后，优先看以下指标：

1. `danger_flash_rate`
2. `safe_flash_rate`
3. `escape_effective_flash_count`
4. `flash_leave_danger_count`
5. `wall_cross_effective_rate`
6. `avg_flash_distance_gain`
7. `avg_flash_min_margin_gain`
8. `avg_flash_openness_gain`

### 7.1 接入后期望现象

理想现象是：

- flash 仍主要发生在 danger / near-threat
- safe flash 不明显反弹
- wall_cross 能力继续保留
- flash 后距离、最小安全裕量、开阔度都继续为正增益

### 7.2 接入后要警惕的回退

以下现象说明接法有问题：

1. `safe_flash_rate` 明显反弹
2. `danger_flash_rate` 明显下降
3. `wall_cross_effective_rate` 明显回退
4. `avg_steps` 或整体逃生表现下降

## 8. 给接入线程的最简结论

如果另一个 Thread 只看一段话，就看下面这段：

```text
当前已验证成功的 flash 策略，不是 true threat 内的硬脱险，而是 near-threat / danger 阶段的预防式脱险。

它最稳定的能力是：
1. 怪物逼近时主动 flash
2. 大量利用穿墙换边
3. 闪后更远、更安全、更开阔

推荐接法：
不要替换队友的主策略，只把当前 flash planner 作为独立闪现决策层接进去。

接入时优先保留：
danger / near-threat 触发
实际落点评估
leave_danger
distance_gain
min_margin_gain
openness_gain
wall_cross

当前不要把 leave_threat 当作主验收目标，也不要把资源 flash 混进来。
```
